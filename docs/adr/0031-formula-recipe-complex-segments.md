# Formula recipes use complex expressions and ordered segments

Formula-authored arbitrary waveform assets use a single SymPy-compatible expression per segment; real-valued output maps to the I channel and complex-valued output maps real/imaginary parts to I/Q. A formula recipe is an ordered list of formula segments, each with its own duration and expression, with one segment as the default case. Empty segment lists are invalid; each segment must have `duration > 0` and a formula string whose `strip()` result is non-empty. Validation and parsing use the stripped expression, but persisted recipe JSON preserves the original formula string so load-and-modify keeps user or agent formatting. The persisted recipe JSON stays minimal: it contains required `segments` and required `normalize`, but no `schema_version`, `kind`, `time_unit`, or `notes` fields. `normalize` only accepts `"none"` or `"peak"` and missing `normalize` is invalid; new GUI recipes use hidden default `"peak"` and preserve the existing normalize value when loading an existing recipe. Durations and time symbols are always interpreted in microseconds, and the only recipe kind in v1 is formula. The total waveform duration is always the sum of segment durations; inserting or deleting a segment changes total duration and never implicitly rescales or redistributes neighboring segment durations. Segment durations do not need to align to the internal render grid, so there is no `duration_not_on_grid` failure. Inside a segment expression, `t` means segment-local time and `T` means whole-waveform global time. Segment rendering uses half-open intervals `[start, end)` for every segment except the final segment, which includes the total-duration endpoint; samples exactly on an internal boundary are evaluated by the following segment. Formula recipes do not expose render sample counts or sample rates; the service chooses `sample_count = round(total_duration * 1000) + 1`, renders a time axis from `0` through `total_duration`, and writes playback arrays into the `.npz` asset before the program layer resamples them to hardware timing. The fixed internal target rate is still bounded by `MAX_ARB_WAVEFORM_SAMPLES = 1_000_001`, so v1 formula-authored waveforms can cover up to about `1000 us` at the internal reference resolution before failing with reason `sample_count_too_large`. First-version formulas do not support conditional constructs such as `Piecewise` or `where`; explicit formula segments are the only supported way to express piecewise waveforms. Formula output defaults to fast-fail when waveform magnitude exceeds normalized range. `normalize: "peak"` is an explicit whole-waveform magnitude-peak normalization mode: compute `peak_abs = max(sqrt(idata**2 + qdata**2))`; if `peak_abs > 0`, divide both I and Q by that same value so `max(Abs) == 1`; if `peak_abs == 0`, leave the all-zero waveform unchanged. This mode also amplifies sub-unit waveforms, rather than only clipping or shrinking over-range output, and its scale factor is derived during rendering or inspection rather than persisted or shown in GUI details. GUI validation disables Save on fast-fail, while agent-facing validation returns structured failure reasons. This avoids parallel I/Q formula fields, cache-resolution knobs, hidden gain changes, duplicate samples at segment boundaries, unbounded preview arrays, and two competing piecewise representations while still supporting shaped multi-part waveforms.
